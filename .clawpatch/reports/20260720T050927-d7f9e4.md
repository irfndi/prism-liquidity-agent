## high: publish_python_pkg skips build when dist/ directory exists but is empty or contains stale artifacts

id: fnd_sig-feat-config-0610d665b5-6274f_b4af893b16
category: build-release
confidence: high
triage: risk
status: open
feature: Shell/workflow config publish-skills.sh (feat_config_0610d665b5)

evidence:
- scripts/publish-skills.sh:64-74 (publish_python_pkg)

Line 64 checks `[ ! -d "dist" ]` to decide whether to run `python -m build`. This only tests directory existence, not whether it contains valid, up-to-date artifacts. Two failure modes result: (1) If `dist/` exists but is empty, the build is skipped and `python -m twine upload dist/*` receives the unexpanded literal glob `dist/*` (bash does not set nullglob here), causing a confusing twine error instead of a build-then-publish flow. (2) If `dist/` exists with stale artifacts from a prior build, the build is silently skipped and outdated packages are uploaded to PyPI, potentially publishing the wrong version. The npm counterpart (line 37) at least checks for a specific file (`dist/index.js`), making it slightly more robust, though it shares the staleness risk.

recommendation:
Always rebuild before publishing, or at minimum check for the presence of actual artifact files (e.g., `dist/*.tar.gz` or `dist/*.whl`) rather than just the directory. A robust pattern: `if [ -z "$(ls -A dist/*.whl dist/*.tar.gz 2>/dev/null)" ]; then python -m build; fi`. For a publish script, unconditionally running the build step is safest.

test analysis:
No tests are included for this feature, and shell publish scripts are rarely unit-tested. The failure requires a pre-existing dist/ directory, which a fresh CI checkout would not have, masking the issue in typical CI runs.

suggested regression test:
Add a shell test (e.g., bats or a simple bash test) that creates an empty `dist/` directory in a fixture package, runs `publish_python_pkg` in dry-run mode, and asserts that the build step is invoked. Repeat with a stale file in `dist/` to verify rebuild.

minimum fix scope:
Change line 64 from `[ ! -d "dist" ]` to a check for actual build artifacts, or unconditionally run `python -m build` before the twine upload.

repro:
mkdir -p packages/langchain-prism/dist && ./scripts/publish-skills.sh  # build is skipped, twine upload fails or uploads stale files

## medium: gh release create stderr suppressed with 2>/dev/null, hiding real failures

category: build-release
confidence: high
triage: risk
status: open
feature: Shell/workflow config release.yml (feat_config_3a72a27053)

evidence:
- .github/workflows/release.yml:306-315

The `2>/dev/null` on the `gh release create` invocation (line 312) suppresses all stderr, including authentication errors, rate-limit messages, and network failures. When create fails for a reason other than 'release already exists', the script falls through to `gh release upload --clobber`, which will also fail but without the original diagnostic context. This makes CI debugging significantly harder and can mask permission/token issues.

recommendation:

test analysis:
No tests exist for this workflow. The suppressed stderr is only observable during actual CI runs with failures.

suggested regression test:
Add a workflow lint step (e.g., actionlint) that flags `2>/dev/null` on commands whose exit status drives control flow.

minimum fix scope:
Remove `2>/dev/null` from line 312 or replace the create-or-upload pattern with an explicit existence check via `gh release view`.

repro:
Temporarily set GITHUB_TOKEN to an invalid value; the create step's error message is swallowed and the upload fallback produces a generic failure.

## medium: bun-version: canary makes builds non-reproducible and vulnerable to upstream breakage

id: fnd_sig-feat-config-3a72a27053-9ae01_ff0edb24c6
category: build-release
confidence: medium
triage: risk
status: open
feature: Shell/workflow config release.yml (feat_config_3a72a27053)

evidence:
- .github/workflows/release.yml:52
- .github/workflows/release.yml:92
- .github/workflows/release.yml:142

All three jobs (test, build-bundles, release) pin Bun to `canary`. Canary builds change daily and are not guaranteed stable. A breaking change in a canary release could simultaneously break testing, building, and releasing. Additionally, because the workflow runs multiple jobs that may start at different times, they could resolve to different canary versions, producing inconsistent artifacts across platforms.

recommendation:
Pin to a specific stable Bun version (e.g., `bun-version: 1.2.15`) or use a lockfile-resolved version. If canary is intentional for testing, at minimum pin the release job to a stable version to ensure reproducible release artifacts.

test analysis:
No tests exercise the CI workflow itself. Reproducibility issues only manifest when the upstream canary changes.

minimum fix scope:
Replace `bun-version: canary` with a pinned stable version string in all three setup-bun steps, or at minimum in the build-bundles and release jobs.

repro:
Wait for a Bun canary release that introduces a breaking change (e.g., in bundler behavior) and push a release tag; all jobs fail or produce inconsistent output.

## medium: Script requires `python` binary which is absent on many modern systems that only provide `python3`

id: fnd_sig-feat-config-0610d665b5-e3499_89fc54985b
category: bug
confidence: medium
triage: risk
status: open
feature: Shell/workflow config publish-skills.sh (feat_config_0610d665b5)

evidence:
- scripts/publish-skills.sh:100-103 (main)
- scripts/publish-skills.sh:67 (publish_python_pkg)
- scripts/publish-skills.sh:74 (publish_python_pkg)

Lines 67 and 74 invoke `python -m build` and `python -m twine upload`, and line 100 gates on `command -v python`. Since macOS 12.3+ and recent Ubuntu/Debian releases, only `python3` is installed by default; the unversioned `python` binary does not exist. On such systems the prerequisite check fails and the script exits, even though a fully functional Python 3 toolchain is available. The project metadata confirms Python is a detected language, so this script is expected to run in environments with Python.

recommendation:
Use `python3` instead of `python`, or add a fallback: `PYTHON="$(command -v python3 || command -v python || true)"` and check that `$PYTHON` is non-empty. Update the prerequisite check accordingly.

test analysis:
No tests are included for this feature. CI environments often have `python` symlinked or aliased, so this would pass in CI while failing on developer machines.

suggested regression test:
Run the script in a container or environment where only `python3` exists (no `python` symlink) and verify the dry-run path completes successfully.

minimum fix scope:
Replace `python` with `python3` on lines 67, 74, and 100, or introduce a `PYTHON` variable resolved from `python3` with a `python` fallback.

repro:
On macOS 12.3+ or Ubuntu 22.04 without python-is-python3: ./scripts/publish-skills.sh --dry-run  # exits with 'python is required but not installed.'

## low: Asset collection silently swallows copy errors with 2>/dev/null || true

id: fnd_sig-feat-config-3a72a27053-ff340_a701ef3cb5
category: build-release
confidence: medium
triage: risk
status: open
feature: Shell/workflow config release.yml (feat_config_3a72a27053)

evidence:
- .github/workflows/release.yml:196-201

Lines 198-199 use `cp ... 2>/dev/null || true` to copy bundle artifacts. While the glob may legitimately match nothing if a directory is empty, this pattern also hides real errors such as disk-full, permission denied, or I/O failures. The downstream manifest completeness check (line 218) partially mitigates missing bundles, but a partial copy (e.g., truncated file due to disk pressure) would pass the count check while shipping a corrupt asset.

recommendation:
Remove `2>/dev/null` and handle the no-match case explicitly: use `find` or check glob expansion (`shopt -s nullglob`) so that a missing file is not an error, but a real copy failure still triggers `set -e`.

test analysis:
No tests cover the workflow. The error suppression is only observable under I/O failure conditions in CI.

minimum fix scope:
Replace `2>/dev/null || true` with `shopt -s nullglob` before the loop and remove the error suppression so real cp failures propagate.

repro:
Fill the runner disk nearly full so that `cp` fails mid-write; the error is suppressed and a truncated tarball ends up in release-assets.

## low: Symlink resolution loop has no cycle guard and can spin indefinitely

id: fnd_sig-feat-cli-command-4828cbbd69-_f06b8dee81
category: bug
confidence: medium
triage: risk
status: open
feature: CLI command prism (feat_cli-command_4828cbbd69)

evidence:
- scripts/prism.sh:10-14

The while loop on line 10 follows symlinks iteratively but has no upper bound on iterations. If the symlink chain contains a cycle (e.g. A → B → A, which can occur from a misconfigured install or a malicious PATH/bin directory), the loop never terminates, hanging the CLI invocation indefinitely. Standard hardened wrappers add a counter (e.g. max 40 hops) or use `realpath`/`readlink -f` which detect cycles internally.

recommendation:
Add a hop counter (e.g. `local hops=0; while [ -L "$SOURCE" ] && [ $hops -lt 40 ]; do ... hops=$((hops+1)); done`) and exit with an error message if the limit is exceeded, or replace the manual loop with `realpath -- "$SOURCE"` (available on macOS 12.3+ and all Linux).

test analysis:
No tests are included for this feature (tests array is empty), and the symlink-resolution path is only exercised at runtime via the bin wrapper.

suggested regression test:
Create a temporary circular symlink pair, invoke the wrapper, and assert it exits non-zero within a timeout rather than hanging.

minimum fix scope:
Add an iteration cap of ~40 to the while loop on line 10 and emit a diagnostic to stderr before exiting 1 when exceeded.

repro:
ln -s /tmp/prism-b /tmp/prism-a && ln -s /tmp/prism-a /tmp/prism-b && cp scripts/prism.sh /tmp/prism-a && /tmp/prism-a
