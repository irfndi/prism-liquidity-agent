## Context (from 4 parallel explore agents)

### Current Architecture
- Revenue config: REVENUE_SHARE_ENABLED (env, default false), REVENUE_SHARE_OPERATOR_PCT (env, default 0)
- Platform fee rate: hardcoded in TIERS (free=0%, pro=5%, fund=10%) in BOTH engine/revenue-service.ts and cloudflare/workers/api/index.ts
- Fee wallet: fetched from API (GET /v1/fee-wallet) with 30-min in-memory cache
- claimFees(): TWO separate on-chain transactions (claim from Meteora SDK, then SPL transfer to fee wallet)
- Revenue reporting: fire-and-forget POST to /v1/revenue/log (no auth, no validation)
- Config loaded once at boot via Effect.runSync(ConfigLive) in engine/index.ts
- Private key NEVER leaves machine (confirmed across all code paths)
- Paper trading default; live mode requires wallet

### Key Gaps Found
1. Config is client-side (env vars) — user can set REVENUE_SHARE_ENABLED=false to bypass
2. Two separate transactions for claim+transfer — user can intercept between them
3. Transfer failure is swallowed (catchAll returns undefined) — fees retained by user
4. No server-side validation of reported revenue — API accepts any numbers
5. No local audit trail of fee claims — only fire-and-forget API log

## Learnings (patterns / pitfalls for next turn)
- Effect-TS pattern: Context.Tag + Layer.provide chain in buildLayer()
- SQLite pattern: MIGRATIONS array in db.ts, versioned sequential migrations
- API auth pattern: Bearer token -> loginHandler -> userId/tier resolution
- Transaction pattern: Effect.tryPromise for async, Effect.catchAll for fallback
- Service addition pattern: services.ts (Tag) -> x-service.ts (Live) -> program.ts (Layer.provide)
